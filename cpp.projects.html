Large projects in C++

In short

I like

Performance

I feel like we get 80% of the performance for 20% of the effort. Modern compilers will find amazing ways to opimize our code as long as it we don't make it a hairy mess, and it's easy enough to measure and control memory allocations.

Libraries

The STL has a good selection of containers, algorithms, and type traits. There are old, stable, and documented libraries like [Boost](), [QuickFIX](), [googletest]()... There are new, good quality libraries for a number of common tasks, like [JSON](), [fmt](), [Catch2](), [](pyBind11)...

Tools

Compilers have improved warnings a lot, and sanitizers are very good at catching a number of bugs early on. A simple CI setup can now help produce programs of a much higher quality than just a few years ago.

Best practices

I don't like

Old habits die hard

  • New compilers have bugs.
  • Modern C++ is juste more complex for no benefit.
  • Clang is slow and doesn't optimize well.
  • GCC generates the best output with `-march=nocona`

There are still large C++ projects stuck on whatever comes with RHEL 5 or 6 (gcc 4.4.X). Anything below 4.8 is painful to use since it won't even implement most of C++11. It would be nice to have a clear governance when it comes to language/implementation versions.

Debugging

Debugging C++ can mean stepping through a large amount of boilerplate generic code and/or macros, espcially when using libraries like the STL and Boost.

Modern C++ constructs like "for (auto& x : xs)" look pretty, but they destroy the ability to single step in debug mode without template hell
